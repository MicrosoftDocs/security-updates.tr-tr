Security Bulletin

Microsoft Güvenlik Bülteni MS10-085 - Önemli
============================================

SChannel Güvenlik Açığı Hizmet Reddine Olanak Verebilir (2207566)
-----------------------------------------------------------------

Yayım Tarihi: 12 Ekim 2010 Salı | Güncelleştirme Tarihi: 18 Ekim 2010 Pazartesi

**Sürüm:** 2.0

### Genel Bilgiler

#### Yürütmeyle İlgili Özet

Bu güvenlik güncelleştirmesi, Windows'daki Güvenli Kanal (SChannel) güvenlik paketindeki özel olarak bildirilen bir güvenlik açığını giderir. Güvenlik açığı, etkilenen bir sistem Güvenli Yuva Katmanı (SSL) aracılığıyla özel hazırlanmış bir paket iletisi alırsa hizmet reddine olanak verebilir. Windows Vista, Windows Server 2008, Windows 7 ve Windows Server 2008 R2'nin tüm desteklenen sürümleri varsayılan olarak SSL ağ trafiği alacak şekilde yapılandırılmamıştır.

Güvenlik güncelleştirmesi Windows Vista, Windows Server 2008, Windows 7 ve Windows Server 2008 R2'nin tüm desteklenen sürümleri için Önemli olarak derecelendirilmiştir. Daha fazla bilgi için, bu bölümdeki **Etkilenen ve Etkilenmeyen Yazılımlar** alt bölümüne bakın.

Güvenlik güncelleştirmesi, SChannel'ın gelen SSL bağlantılarını işlerken protokol isteklerini işleme biçimini değiştirerek bu güvenlik açığını giderir. Bu güvenlik açığı hakkında daha fazla bilgi için, bir sonraki bölüm olan **Güvenlik Açığı Bilgileri** altında, ilgili güvenlik açığı girdisinin Sık Sorulan Sorular (SSS) alt bölümüne bakın.

**Öneri.** Müşterilerin büyük bir bölümü otomatik güncelleştirmeleri etkinleştirmiş durumdadır ve bu güvenlik güncelleştirmesi otomatik olarak karşıdan yüklenip kurulacağı için herhangi bir işlem yapmaları gerekmemektedir. Otomatik güncelleştirmeleri etkinleştirmemiş olan müşterilerin güncelleştirmeleri el ile denetlemeleri ve bu güncelleştirmeyi yüklemeleri gerekir. Otomatik güncelleştirmelerin özel yapılandırma seçenekleri hakkında bilgi için, bkz. [Microsoft Bilgi Bankası makalesi 294871](http://support.microsoft.com/kb/294871).

Yöneticiler ve kurumsal yüklemeler ya da bu güvenlik güncelleştirmesini el ile yüklemek isteyen son kullanıcılar için, Microsoft, müşterilerinin güncelleştirme yönetim yazılımlarını kullanarak veya [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) hizmeti aracılığıyla güncelleştirmeleri denetleyerek güncelleştirmeyi ilk fırsatta uygulamalarını önerir.

Bu bültenin sonrasında yer alan **Algılama ve Dağıtım Araçları ve Kılavuzu** bölümüne de bakın.

**Bilinen Sorunlar.** Yok

Bu Güvenlik Güncelleştirmesi İle İlgili Sık Sorulan Sorular (SSS)
-----------------------------------------------------------------

<span></span>
**Bu bülten 18 Ekim 2010'da neden yeniden düzenlendi?**   
Microsoft bu bülteni, gelen SSL bağlantılarını alacak biçimde yapılandırılmış olan etkilenen sistemlerde bu güvenlik açığından yararlanılabileceğini açıklığa kavuşturmak üzere yeniden düzeltildi. Bu yalnızca bilgilendirme amaçlı bir değişikliktir. Algılama mantığında veya güvenlik güncelleştirmesi paketlerinde herhangi bir değişiklik yapılmamıştır. Otomatik güncelleştirmeleri etkinleştirmiş olanlar da dahil sistemlerini daha önce başarıyla güncelleştirmiş olan müşterilerin herhangi bir işlem gerçekleştirmeleri gerekmez. Bu güncelleştirmeyi daha önce yüklememiş olan müşterilerin, yeniden düzenlenen güvenlik açığı bilgilerine dayalı olarak sistemleri için bu güncelleştirmenin gerekli olup olmadığını yeniden değerlendirmeleri gerekebilir.

TLSv1 Hizmet Reddi Güvenlik Açığı - CVE-2010-3229  
-------------------------------------------------
  
<span></span>
Windows Server 2008, Windows Vista, Windows Server 2008 R2 ve Windows 7 üzerinde SChannel'ın gelen SSL bağlantılarını işlerken protokol isteklerini işleme biçiminde bir hizmet reddi güvenlik açığı bulunmaktadır. Bir uzak anonim saldırgan etkilenen sisteme özel hazırlanmış bir ağ paketi göndererek, LSASS hizmetinin durdurulmasına ve sistemin yeniden başlatılmasına neden olabilir. Yalnızca bir hizmetin veya uygulamanın SSL ağ trafiği alacak biçimde yapılandırılmış olduğu sistemler bu açıktan etkilenir.
  
Bu güvenlik açığını Ortak Güvenlik Açıkları ve Etkilenmeler listesinde standart girdi olarak görüntülemek için, bkz: [CVE-2010-3229](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-3229).
  
#### TLSv1 Hizmet Reddi Güvenlik Açığı - CVE-2010-3229 için Azaltıcı Etkenler
  
Azaltıcı etken, varolan bir durumda bulunan ve güvenlik açığından yararlanılmasının önem derecesini azaltabilecek bir ayar, genel yapılandırma ya da en iyi uygulama anlamına gelir. Sizin durumunuzda aşağıdaki azaltıcı etkenler işinize yarayabilir:
  
-   Güvenlik açığından yalnızca bir hizmetin veya uygulamanın SSL ağ trafiği alacak biçimde yapılandırılmış olduğu sistemlerde yararlanılabilir. Yalnızca giden SSL bağlantıları başlatan sistemlerde bu güvenlik açığından yararlanılamaz.  
-   Windows Vista, Windows Server 2008, Windows 7 ve Windows Server 2008 R2'nin tüm desteklenen sürümleri varsayılan olarak SSL ağ trafiği alacak şekilde yapılandırılmamıştır.
  
#### TLSv1 Hizmet Reddi Güvenlik Açığı - CVE-2010-3229 için Geçici Çözümler
  
Microsoft bu güvenlik açığı için herhangi bir geçici çözüm belirleyememiştir.
  
#### TLSv1 Hizmet Reddi Güvenlik Açığı - CVE-2010-3229 Hakkında SSS:
  
**Bu güvenlik açığının kapsamı nedir?**  
Bu bir hizmet reddi güvenlik açığıdır. Bu güvenlik açığından yararlanan bir saldırgan LSASS hizmetinin durdurulmasına, bilgisayarın yeniden başlatılmasına ve sistem başarıyla yeniden başlatılıncaya kadar hizmet reddi koşulunun oluşmasına neden olur. Saldırgan daha sonra güvenlik açığından yeniden yararlanabilir.
  
**Güvenlik açığına ne neden olmaktadır?**  
Güvenlik açığı, SChannel'ın SSL/TLS el sıkışması sırasında bir protokol isteğini işlemeden önce bir mantıksal koşulu denetlememesi nedeniyle ortaya çıkar.
  
**SChannel güvenlik paketi nedir?**  
Güvenli Kanal (SChannel) güvenlik paketi, Güvenli Yuva Katmanı (SSL) ve Aktarım Katmanı Güvenliği (TLS) Internet standartları kimlik doğrulama protokollerini kullanan bir Güvenlik Destek Sağlayıcısı'dır (SSP). Bu bileşenler, Web'de gezinme gibi birkaç genel Internet ve ağ uygulamasını desteklemek için güvenli iletişim sağlamak üzere kullanılır. SChannel, istemci ile sunucu arasında güvenli iletişim için kimlik doğrulama hizmeti sağlanmasına yardımcı olan güvenlik paketinin bir parçasıdır.
  
Daha fazla bilgi için, [Güvenli Kanal](http://msdn.microsoft.com/en-us/library/aa380123(vs.85).aspx) adlı MSDN makalesine bakın.
  
**TLS ve SSL nedir?**  
Aktarım Katmanı Güvenliği (TLS) El Sıkışma Protokolü, güvenli oturumlar oluşturmak veya sürdürmek için gereken kimlik doğrulamasından ve anahtar değişiminden sorumludur. Güvenli Yuva Katmanı (SSL), Aktarım Katmanı Güvenliği protokolünün öncülüdür. Hem TLS hem de SSL aynı işlevleri gerçekleştirir ve güvenli ağ iletişimini desteklemek üzere ortak ve gizli anahtar teknolojilerini bir arada kullanır.
  
Daha fazla bilgi için, [TLS/SSL Nasıl Çalışır](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) adlı TechNet makalesine bakın.
  
**TLS veya SSL el sıkışması nedir?**  
TLS veya SSL kullanılarak güvenli bir oturum oluşturulurken El Sıkışma Protokolü şunları yönetir:
  
-   Şifre paketi görüşmesi  
-   Sunucunun ve isteğe bağlı olarak istemcinin kimliğinin doğrulanması  
-   Oturum anahtarı bilgilerinin değişimi
  
Daha fazla bilgi için, [TLS/SSL Nasıl Çalışır](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) adlı TechNet makalesine bakın.
  
**Bir saldırgan bu güvenlik açığından yararlanarak neler yapabilir?**  
Bu güvenlik açığından başarıyla yararlanan bir saldırgan, bir sunucunun yanıt vermemeye başlamasına ve yeniden başlatılması gerekmesine neden olabilir.
  
**Bir saldırgan güvenlik açığından nasıl yararlanabilir?**  
Bir uzak anonim saldırgan, etkilenen bir sistemde çalıştırılan SSL etkin bir hizmete veya uygulamaya özel hazırlanmış bir paket iletisi gönderebilir. Güvenlik açığından yalnızca hizmet veya uygulama SSL trafiğini işlemek için SChannel kullanıyorsa yararlanılabilir. SSL etkin hizmet veya uygulamalara örnek olarak Internet Information Services (IIS), Uzak Masaüstü Protokolü (RDP) ve Güvenli Yuva Tünel Protokolü'nün (SSTP) yanı sıra SSL üzerinden SMTP ya da SSL üzerinden LDAP gibi diğer tünel oluşturma uygulamaları da gösterilebilir. Tüm bu durumlarda, saldırgan SSL el sıkışmasının bir parçası olarak özel hazırlanmış bir paket gönderebilir. Bu açıktan başarıyla yararlanılması, LSASS hizmetinin yanıt vermemeye başlamasına ve sistemin yeniden başlatılmasına yol açarak geçici bir hizmet reddi durumuna neden olur. Sistem başarıyla yeniden başlatıldıktan sonra saldırgan bu güvenlik açığından tekrar yararlanabilir.
  
**Bu güvenlik açığından birincil derecede hangi sistemler etkileniyor?**  
Bu güvenlik açığından, bir uygulamanın veya hizmetin gelen SSL bağlantılarını alacak biçimde yapılandırılmış olduğu etkilenen Microsoft Windows sürümlerini çalıştıran sistemlerde yararlanılabilir. Etkilenen sistemlere örnek olarak SSL bağlantılarını kabul eden Web sunucuları olarak yapılandırılmış olan sistemler, gelen SSTP bağlantılarını kabul eden VPN sunucuları ve Uzak Masaüstü Hizmetleri rolü atanmış olan sunucular gösterilebilir. Uzak Masaüstü Protokolü'nün /RDP) etkinleştirilmiş olduğu etkilenen bir Microsoft Windows sürümü çalıştırılan iş istasyonları da risk altındadır.
  
SSL bağlantıları başlatan ama bu tür bağlantılar almayan sistemler daha az risk altındadır. Daha az risk altında olan sistemlere örnek olarak, kullanıcıların Web'e gözattığı iş istasyonları veya kullanıcıların bir VPN'ye istemci olarak bağlandığı dizüstü bilgisayarlar gösterilebilir. Bu tür istemci sistemlerde de gelen SSL bağlantılarını kabul eden uygulamalar veya hizmetler olabileceğini ve bu durumda bu sistemlerin risk altında kalacağını unutmayın.
  
**Güncelleştirme ne yapıyor?**  
Güncelleştirme, SChannel'ın gelen SSL bağlantılarını işlerken protokol isteklerini işleme biçimini değiştirerek bu güvenlik açığını giderir.
  
**Bu güvenlik bülteni yayımlandığında bu güvenlik açığı genel olarak duyurulmuş muydu?**  
Hayır. Microsoft, bu güvenlik açığı hakkında bilgilere koordinasyon içinde olduğu sorumlu kişiler aracılığıyla ulaştı.
  
**Bu güvenlik bülteni yayımlandığında, Microsoft bu güvenlik açığından yararlanıldığına dair herhangi bir rapor almış mıydı?**  
Hayır. Bu güvenlik bülteni ilk yayımlandığında, Microsoft bu güvenlik açığının müşterilere saldırmak üzere genel olarak kullanıldığına dair herhangi bir bilgi almamıştı.
