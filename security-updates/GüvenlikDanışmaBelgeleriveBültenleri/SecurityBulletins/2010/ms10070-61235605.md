Security Bulletin

Microsoft Güvenlik Bülteni MS10-070 - Önemli
============================================

ASP.NET'teki Güvenlik Açığı Bilginin Açığa Çıkmasına Neden Olabilir (2418042)
-----------------------------------------------------------------------------

Yayım Tarihi: 28 Eylül 2010 Salı | Güncelleştirme Tarihi: 26 Ekim 2011 Çarşamba

**Sürüm:** 4.2

### Genel Bilgiler

#### Yürütmeyle İlgili Özet

Bu güvenlik güncelleştirmesi ASP.NET'teki genel olarak duyurulan bir güvenlik açığını giderir. Güvenlik açığı, bilginin açığa çıkmasına neden olabilir. Bu güvenlik açığından başarıyla yararlanan bir saldırgan, sunucu tarafından şifrelenen görünüm durumu gibi verileri görüntüleyebilir. Bu güvenlik açığı, veri değiştirme amacıyla da kullanılabilir; güvenlik açığından başarıyla yararlanıldığında, sunucu tarafından şifrelenen verilerin şifresi çözülebilir ve veriler değiştirilebilir. Microsoft .NET Framework 3.5 Service Pack 1'den önceki Microsoft .NET Framework sürümleri, bu güvenlik açığının dosya içeriğini açığa çıkarma bölümünden etkilenmez.

Bu güvenlik güncelleştirmesi Microsoft .NET Framework 1.0 Service Pack 3 dışında tüm desteklenen ASP.NET sürümleri için Önemli olarak derecelendirilmiştir. Daha fazla bilgi için, bu bölümdeki **Etkilenen ve Etkilenmeyen Yazılımlar** alt bölümüne bakın.

Güvenlik güncelleştirmesi, ASP.NET tarafından şifrelenen tüm verilerde ek imza kullanarak bu güvenlik açığını giderir. Bu güvenlik açığı hakkında daha fazla bilgi için, bir sonraki bölüm olan **Güvenlik Açığı Bilgileri** altında, ilgili güvenlik açığı girdisinin Sık Sorulan Sorular (SSS) alt bölümüne bakın.

Bu güvenlik güncelleştirmesi ilk olarak [Microsoft Güvenlik Danışma Belgesi 2416728](http://technet.microsoft.com/security/advisory/2416728)'de açıklanan güvenlik açığını da giderir.

**Öneri.** Microsoft, müşterilerin güncelleştirmeyi ilk fırsatta uygulamalarını önerir.

Bu bültenin sonrasında yer alan **Algılama ve Dağıtım Araçları ve Kılavuzu** bölümüne de bakın.

**Bilinen Sorunlar.**[Microsoft Bilgi Bankası makalesi 2418042](http://support.microsoft.com/kb/2418042), bu güvenlik güncelleştirmesini yükleyen müşterilerin karşılaşabileceği bilinen sorunları belgelemektedir. Bu makale ayrıca, bu sorunlar için önerilen çözümleri de içermektedir.

#### Etkilenen ve Etkilenmeyen Yazılımlar

Aşağıdaki yazılımlar, hangi sürümlerinin etkilendiğini belirlemek amacıyla sınanmıştır. Diğer sürümler destek ömrünü tamamlamıştır veya bu açıktan etkilenmemektedir. Yazılımınızın sürümünün destek ömrünü belirlemek için [Microsoft Destek Ömrü](http://go.microsoft.com/fwlink/?linkid=21742) Web sitesini ziyaret edin.

Bu Güvenlik Güncelleştirmesi İle İlgili Sık Sorulan Sorular (SSS)
-----------------------------------------------------------------

<span></span>
**Bu bülten 22 Şubat 2011'de neden yeniden düzenlendi?**  
Microsoft bu güvenlik bültenini, 32-bit sistemler için Windows 7 Service Pack 1, x64 tabanlı sistemler için Windows 7 Service Pack 1, x64 tabanlı sistemler için Windows Server 2008 R2 Service Pack 1 ve Itanium tabanlı sistemler için Windows Server 2008 R2 Service Pack 1 çalıştırılan sistemlere Microsoft .NET Framework 4.0 (KB2416472) güncelleştirme paketlerinin sunulduğu bir algılama değişikliğini duyurmak üzere yeniden düzenledi. Bu algılama değişikliği yalnızca 32-bit sistemler için Windows 7 Service Pack 1, x64 tabanlı sistemler için Windows 7 Service Pack 1, x64 tabanlı sistemler için Windows Server 2008 R2 Service Pack 1 veya Itanium tabanlı sistemler için Windows Server 2008 R2 Service Pack 1'i yükledikten sonra Microsoft .NET Framework 4.0'ı yüklemiş olan müşteriler için geçerlidir. Sistemlerini daha önce başarıyla güncelleştirmiş olan müşterilerin herhangi bir işlem gerçekleştirmeleri gerekmez.

**Bu bülten 14 Aralık 2010'da neden yeniden düzenlendi?**  
Microsoft bu güvenlik bültenini, Microsoft .NET Framework 4.0'a yönelik yeni güncelleştirme paketlerinin (KB2416472) kullanılabilir olduğunu duyurmak üzere yeniden düzenledi. Bu yeni paketlerde, diğer güncelleştirmelerin başarıyla yüklenmesini engelleyebilecek bir kurulum sorunu düzeltilmiştir. Bu sorundan etkilenmiş olabilecek başka bir ürün ya da güncelleştirme yüklemesi olan müşteriler ek bilgi için [Microsoft Bilgi Bankası makalesi 2473228](http://support.microsoft.com/kb/2473228)'e bakabilirler. Sistemlerini daha önce başarıyla güncelleştirmiş olan müşterilerin herhangi bir işlem gerçekleştirmeleri gerekmez.

**.NET Framework 3.0 Service Pack 2 yüklü; bu sürüm, bu bültende açıklanan etkilenen yazılımlar arasında listelenmiyor. Güncelleştirme yüklemem gerekiyor mu?**  
Bu bültende .NET Framework 2.0 ve .NET Framework 3.5 özellik katmanlarındaki bir güvenlik açığı açıklanır. .NET Framework 3.0 Service Pack 2 yükleyicisi, .NET Framework 2.0 Service Pack 2 kurulumu için zincir oluşturarak bir sürümün daha önceki sürümü de yüklemesini sağlar. Bu nedenle, .NET Framework 3.0 Service Pack 2'nin yüklü olduğu müşterilerin .NET Framework 2.0 Service Pack 2 güvenlik güncelleştirmelerini yüklemeleri gerekmektedir.

**.NET Framework 3.5 yüklü. Başka bir güncelleştirme yüklemem gerekiyor mu?**  
Bu bültende .NET Framework 2.0 ve .NET Framework 3.5 özellik katmanlarındaki bir güvenlik açığı açıklanır. .NET Framework 3.5 yükleyicisi hem .NET Framework 2.0 Service Pack 1 kurulumunu hem de .NET Framework 3.0 Service Pack 1 kurulumunu art arda yükler. Bu nedenle, .NET Framework 3.5'i yüklemiş olan müşteriler .NET Framework 3.5 güncelleştirmelerine ek olarak .NET Framework 2.0 Service Pack 1 güvenlik güncelleştirmelerini de yüklemelidir.

Sisteminizde başka bir .NET Framework sürümünün yüklü olup olmadığını belirlemenize yardımcı olması için, bu bölümün sonrasındaki SSS bölümünde yer alan "Microsoft .NET Framework'ün hangi sürümünün yüklü olduğunu nasıl belirlerim?" girdisine bakın.

**.NET Framework 3.5 Service Pack 1 yüklü. Başka bir güncelleştirme yüklemem gerekiyor mu?**  
Bu bültende .NET Framework 2.0 ve .NET Framework 3.5 özellik katmanlarındaki bir güvenlik açığı açıklanır. .NET Framework 3.5 Service Pack 1 yükleyicisi hem .NET Framework 2.0 Service Pack 2 kurulumunu hem de .NET Framework 3.0 Service Pack 2 kurulumunu art arda yükler. Bu nedenle, .NET Framework 3.5 Service Pack 1'in yüklü olduğu müşterilerin .NET Framework 2.0 Service Pack 2 güvenlik güncelleştirmelerini de yüklemeleri gerekmektedir.

Sisteminizde başka bir .NET Framework sürümünün yüklü olup olmadığını belirlemenize yardımcı olması için, bu bölümün sonrasındaki SSS bölümünde yer alan "Microsoft .NET Framework'ün hangi sürümünün yüklü olduğunu nasıl belirlerim?" girdisine bakın.

**Bu bülten 30 Eylül 2010'da neden yeniden düzenlendi?**

ASP.NET'te Doldurma Bilgisi Güvenlik Açığı - CVE-2010-3332
----------------------------------------------------------

<span></span>
ASP.NET'te, şifreleme doldurma doğrulaması sırasında hata işlemenin yanlış yapılması nedeniyle bilginin açığa çıkması güvenlik açığı bulunmaktadır. Bu güvenlik açığından başarıyla yararlanan bir saldırgan, sunucu tarafından şifrelenen görünüm durumu gibi verileri görüntüleyebilir. Bu güvenlik açığı, veri değiştirme amacıyla da kullanılabilir; güvenlik açığından başarıyla yararlanıldığında, sunucu tarafından şifrelenen verilerin şifresi çözülebilir ve veriler değiştirilebilir. Bu güvenlik açığının, bir saldırganın doğrudan kod yürütmesine veya kullanıcı haklarını yükseltmesine izin vermeyeceğini unutmayın; ancak, etkilenen sistemden daha fazla yararlanmak amacıyla kullanılacak bilgileri üretmek için kullanılabilir. Microsoft .NET Framework 3.5 Service Pack 1 ve sonrasında bu güvenlik açığı, bir saldırgan tarafından web.config dosyası da dahil ASP.NET uygulamasındaki herhangi bir dosyanın içeriğini elde etmek için de kullanılabilir.

Bu güvenlik açığını Ortak Güvenlik Açıkları ve Etkilenmeler listesinde standart girdi olarak görüntülemek için, bkz [CVE-2010-3332](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-3332).

#### ASP.NET'te Doldurma Bilgisi Güvenlik Açığı - CVE-2010-3332 için Azaltıcı Etkenler

Azaltıcı etken, varolan bir durumda bulunan ve güvenlik açığından yararlanılmasının önem derecesini azaltabilecek bir ayar, genel yapılandırma ya da en iyi uygulama anlamına gelir. Sizin durumunuzda aşağıdaki azaltıcı etkenler işinize yarayabilir:

-   Microsoft .NET Framework 3.5 Service Pack 1'den önceki Microsoft .NET Framework sürümleri, bu güvenlik açığının dosya içeriğini açığa çıkarma bölümünden etkilenmez.

#### ASP.NET'te Doldurma Bilgisi Güvenlik Açığı - CVE-2010-3332 için Geçici Çözümler

Geçici çözüm, arka plandaki güvenlik açığını gidermeyen ama güvenlik açığı uygulanmadan önceki bilinen saldırı vektörlerinin engellenmesine yardımcı olan bir ayar veya yapılandırma değişikliği anlamına gelir. Microsoft aşağıdaki geçici çözümleri sınamıştır ve bir geçici çözümün işlevselliği azaltıp azaltmadığı ilgili açıklamada belirtilmektedir:

-   **UrlScan veya İstek Filtreleme kuralını etkinleştirin, ASP.NET özel hatalarını etkinleştirin ve tüm hata kodlarını aynı hata sayfasıyla eşleyin**

    ASP.NET'in customErrors özelliğini etkinleştirip sunucuda hangi hata oluşursa oluşsun uygulamaları her zaman aynı hata sayfasını döndürecek şekilde yapılandırmak, geçerli yararlanmayı kullanan bir saldırganın sunucuda oluşan farklı hata türlerini ayırt etmesini zorlaştırabilir.

    .NET Framework 3.5 Service Pack 1 veya ve sonraki sürümlerinin kullanıldığı sistemlerde bu geçici çözüm, geçerli yararlanmanın zamanla ilgili saldırı vektörüne karşı da koruma sağlar. Geçici çözüm, customErrors özelliğinde redirectMode="ResponseRewrite" seçeneğini kullanır ve hata sayfasında rasgele bir gecikme görüntüler. Bu yaklaşımlar bir arada kullanılarak, saldırganın hatayı alırken geçen zamanı ölçerek sunucuda oluşan hata türünü tahmin etmesini zorlaştırır.

    Bu geçici çözüm ayrıca sorgu dizesinde uygulama hatası yolunu belirten isteklerin engellenmesini gerektirir. Bu işlem, ücretsiz bir Internet Information Services (IIS) aracı olan ve yönetici tarafından tanımlanan kurallara dayalı olarak istekleri seçmeli olarak engelleyebilen URLScan kullanılarak gerçekleştirilebilir. Windows Vista Service Pack 2, Windows Server 2008 Service Pack 2, Windows 7 veya Windows Server 2008 R2 üzerinde Internet Information Services (IIS) çalıştırıyorsanız, İstek Filtreleme özelliğini de kullanabilirsiniz.

    **İsteğin sorgu dizesinde ASP.Net uygulama hatası yolunu değiştiren istekleri engelleyin**

    **UrlScan kullanarak:**

    1. [UrlScan 3.1](http://www.iis.net/download/urlscan)'i karşıdan yükleyin ve kurun. UrlScan'i yapılandırma ve kullanma hakkında daha ayrıntılı yönergeler için, bkz. [UrlScan 3 Başvurusu](http://learn.iis.net/page.aspx/938/urlscan-3-reference/).

    2. UrlScan.ini dosyasını değiştirin (**%windir%\\system32\\inetsrv\\urlscan** dizininde bulunmaktadır). Aşağıdaki satırı, Urlscan.ini dosyasındaki **\[DenyQueryStringSequences\]** bölümünün altına ekleyin:

    `aspxerrorpath=`

    Bunu yaptıktan sonra, **\[DenyQueryStringSequences\]** bölümü aşağıdakine benzer olmalıdır (bölümde ek satır bulunabilir ve bu durum geçici çözümü etkilemez):

    `[DenyQueryStringSequences]aspxerrorpath=`

    3. Yönetici olarak oturum açmış durumdayken komut isteminden **iisreset** komutunu çalıştırın.

    **IIS istek filtreleme kullanarak:**

    Bu yönergeler Windows Vista Service Pack 2, Windows Server 2008 Service Pack 2, Windows 7 veya Windows Server 2008 R2 üzerinde IIS çalıştırılan sistemler için yukarıdaki UrlScan yönergelerinin alternatifleridir.

    1. **Internet Information Services, World Wide Web Hizmetleri, Güvenlik** altından İstek Filtreleme özelliğini seçerek, Program Ekle/Kaldır veya Rol Yöneticisi aracılığıyla bu özelliği yükleyin.

    2. **Internet Information Services (IIS) Yöneticisi**'ni başlatın.

    3. Soldaki bölmede sunucu düğümünü seçin.

    4. **İstek Filtreleme**'yi çift tıklatın.

    5. **Sorgu Dizeleri** sekmesini seçin ve **Eylemler** bölmesinden **Sorgu Dizesini Reddet …** komutunu tıklatın.

    6. İletişim kutusuna **aspxerrorpath=** değerini girin ve **Tamam**'ı seçin.

    Alternatif olarak, aşağıdaki appcmd komutunu kullanarak da bu istek sorgu dizesini ayarlayabilirsiniz:

    `appcmd set config /section:requestfiltering /+denyQueryStringSequences.[sequence='aspxerrorpath=']`

    Appcmd komutunu kullanarak IIS'yi yapılandırma hakkında daha fazla bilgi için, bkz. [AppCmd.exe ile Çalışmaya Başlayın](http://learn.iis.net/page.aspx/114/getting-started-with-appcmdexe).

    **ASP.Net uygulamalarını tekdüzen özel hatalar kullanacak biçimde yapılandırın**

    Her bir ASP.NET web uygulamasının kök klasöründe bir web.config dosyasının bulunup bulunmadığını belirleyin. Bu geçici çözümü uygulamak için hedef dizinde dosya oluşturma izinlerine sahip olmanız gerekir.

    ASP.NET uygulamasında bir web.config dosyası yoksa:

    **.NET Framework 3.5 ve öncesinde**

    1. ASP.NET uygulamasının kök klasöründe **web.config** adlı bir metin dosyası oluşturun ve aşağıdaki içeriği ekleyin:

    `<configuration><location allowOverride="false"><system.web><customErrors mode="On" defaultRedirect="~/error.html" /></system.web></location></configuration>`

    2. Genel bir hata iletisi içeren **error.html** adlı bir metin dosyası oluşturun ve ASP.NET uygulamasının kök klasörüne kaydedin.

    3. Alternatif olarak, **web.config** dosyasındaki error.html sayfasını varolan bir hata sayfasına işaret edecek şekilde yeniden adlandırabilirsiniz; ancak bu sayfanın bağlama özel içerik değil, genel içerik görüntülemesi gerekir.

    **.NET Framework 3.5 Service Pack 1 ve sonrasında**

    1. ASP.NET uygulamasının kök klasöründe **web.config** adlı bir metin dosyası oluşturun ve aşağıdaki içeriği ekleyin:

    `<configuration><location allowOverride="false"><system.web><customErrors mode="On" redirectMode="ResponseRewrite" defaultRedirect="~/ErrorPage.aspx" /></system.web></location></configuration>`

    2. C\# kullanmayı biliyorsanız, aşağıdaki **ErrorPage.aspx** dosyasını kullanmanız önerilir:

    `<%@ Page Language="C#" AutoEventWireup="true" %><%@ Import Namespace="System.Security.Cryptography" %><%@ Import Namespace="System.Threading" %><script >void Page_Load() {byte[] delay = new byte[1];RandomNumberGenerator prng = new RNGCryptoServiceProvider();prng.GetBytes(delay);Thread.Sleep((int)delay[0]);IDisposable disposable = prng as IDisposable;if (disposable != null) { disposable.Dispose(); }}</script><!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd"><html xmlns="http://www.w3.org/1999/xhtml"><head ><title></title></head><body><div>İsteğiniz işlenirken bir hata meydana geldi.</div></body></html>`

    3. Visual Basic .NET kullanmayı biliyorsanız, aşağıdaki **ErrorPage.aspx** dosyasını kullanmanız önerilir:

    `<%@ Page Language="VB" AutoEventWireup="true" %><%@ Import Namespace="System.Security.Cryptography" %><%@ Import Namespace="System.Threading" %><script >Sub Page_Load()Dim delay As Byte() = New Byte(0) {}Dim prng As RandomNumberGenerator = New RNGCryptoServiceProvider()prng.GetBytes(delay)Thread.Sleep(CType(delay(0), Integer))Dim disposable As IDisposable = TryCast(prng, IDisposable)If Not disposable Is Nothing Thendisposable.Dispose()End IfEnd Sub</script><!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd"><html xmlns="http://www.w3.org/1999/xhtml"><head ><title></title></head><body><div>İsteğiniz işlenirken bir hata meydana geldi.</div></body></html>`

    ASP.NET uygulamasında bir **web.config** dosyası varsa:

    **.NET Framework 3.5 RTM ve öncesinde**

    1. Aşağıdaki örnekte köşeli parantez içindeki metni varolan **web.config** dosyanıza ekleyin:

    `<?xml version="1.0"?><configuration><configSections>...</configSections><appSettings>...</appSettings><connectionStrings>...</connectionStrings>[<location allowOverride="false"><system.web><customErrors mode="On" defaultRedirect="~/error.html" /></system.web></location>]<system.web>...</system.web><system.codedom>...</system.codedom></configuration>`

    2. Genel bir hata iletisi içeren **error.html** adlı bir metin dosyası oluşturun ve ASP.NET uygulamasının kök klasörüne kaydedin.

#### ASP.NET'te Doldurma Bilgisi Güvenlik Açığı - CVE-2010-3332 Hakkında SSS

**Bu güvenlik açığının kapsamı nedir?**  
Bu, bir bilginin açığa çıkması güvenlik açığıdır. Bu güvenlik açığından başarıyla yararlanan bir saldırgan, sunucu tarafından şifrelenen görünüm durumu gibi verileri görüntüleyebilir. Bu güvenlik açığı, veri değiştirme amacıyla da kullanılabilir; güvenlik açığından başarıyla yararlanıldığında, sunucu tarafından şifrelenen verilerin şifresi çözülebilir ve veriler değiştirilebilir. Microsoft .NET Framework 3.5 Service Pack 1 ve sonrasında bu güvenlik açığı, bir saldırgan tarafından web.config dosyası da dahil ASP.NET uygulamasındaki herhangi bir dosyanın içeriğini elde etmek için de kullanılabilir. Bu güvenlik açığının, bir saldırganın doğrudan kod yürütmesine veya kullanıcı haklarını yükseltmesine izin vermeyeceğini unutmayın; ancak, etkilenen sistemden daha fazla yararlanmak amacıyla kullanılacak bilgileri üretmek için kullanılabilir.

**Güvenlik açığına ne neden olmaktadır?**  
Şifreleme doldurmanın ASP.NET tarafından kullanılması, hata yanıtlarındaki şifreli verileri okumak ve değiştirmek üzere saldırgan tarafından kullanılabilecek bilgiler sağlar.

**ASP.NET nedir?**  
ASP.NET, Microsoft .NET Framework'te bulunan ve geliştiricilerin Web uygulamaları ve XML Web Hizmetleri oluşturmasına olanak sağlayan bir grup teknolojidir.

Statik HTML ve komut dosyalarını bir arada kullanan geleneksel Web sayfalarından farklı olarak, ASP.NET derlenmiş ve olay kaynaklı sayfalar kullanır. ASP.NET Web tabanlı bir uygulama ortamı olduğu için, temel HTTP işlevselliğini sağlamak üzere arka planda bir Web sunucusunun bulunması gerektiğinden Internet Information Services'ın (IIS) üzerinde çalışır. Daha fazla bilgi için, bkz: [Resmi Microsoft ASP.NET Sitesi](http://www.asp.net/).

**ASP.NET Görünüm Durumu nedir?**  
Microsoft ASP.NET Görünüm Durumu, geri göndermeler sırasında ASP.NET Web sayfası tarafından Web Formu'nun durumundaki değişiklikleri korumak için kullanılan tekniktir. Bir sayfanın Görünüm Durumu varsayılan olarak **\_\_VIEWSTATE** adlı gizli bir forma yerleştirilir. ASP.NET Görünüm Durumu hakkında daha fazla bilgi için, [ASP.NET Görünüm Durumu'nu Anlama](http://msdn.microsoft.com/en-us/library/ms972976.aspx) adlı MSDN makalesine bakın.

**Bir saldırgan bu güvenlik açığından yararlanarak neler yapabilir?**  
Bu güvenlik açığından başarıyla yararlanan bir saldırgan, sunucu tarafından şifrelenen görünüm durumu gibi verileri görüntüleyebilir. Böylece saldırgan web sitesi dizinindeki veya alt dizinlerindeki web.config gibi herhangi bir dosyanın içeriğini okuyabilir. Web.config dosyasında genellikle önemli bilgiler depolanır. Bu tür bilgilerin açığa çıkmasında oluşacak sonuçlar, bilginin yapısına bağlıdır.

Bu güvenlik açığından başarıyla yararlanan bir saldırgan ayrıca hedef sunucuda web sitesi dizininde ya da alt dizinlerde bulunan dosyalardaki verileri de okuyabilir.

**Bir saldırgan bu güvenlik açığından nasıl yararlanabilir?**  
Saldırgan bu güvenlik açığından yararlanmak için, etkilenen bir sunucuya bir Web isteği aracılığıyla şifre metni gönderir ve web sitesinin döndürdüğü hata kodunu inceleyerek metnin şifresinin doğru çözülmüş olup olmadığını belirler. Bu isteklerden yeterli sayıda gönderen bir saldırgan, şifrelenmiş verileri okumak ya da değiştirmek için yeterli bilgiye sahip olabilir.

**Bu güvenlik açığından birincil derecede hangi sistemler etkileniyor?**  
ASP.NET'in yüklü olduğu Web sunucuları bu güvenlik açığına karşı risk altındadır.

**Aşağıda listelenen geçici çözümü kullanmak yerine bir hata sayfasına yeniden yönlendiren özel bir günlük modülü kullanıyorsam bu güvenlik açığından etkilenir miyim?**  
Özel günlük modülünüzden gönderilen yanıtlar istemcinin hata yanıtlarını içeriğine ya da hizmet verilme zamanına göre ayırt etmesine olanak vermiyorsa, bu modül customErrors geçici çözümü yerine kullanılabilir. Bu yanıtlar hem HTTP yanıtının tamamını hem de HTTP hata kodunu içerir. Yukarıda belirtilenler her zaman geçerli değilse bu durum yeterli değildir.

**Bu soruna karşı koruma sağlamak üzere özel bir 404 sayfası ve tüm diğer hatalar için varsayılan bir yeniden yönlendirme oluşturabilir miyim?**  
Hayır. Saldırgan 404 hatasıyla diğer hataları yine de birbirinden ayırabilir. Bu saldırıya karşı korunmak için hataları homojen hale getirmek oldukça önemli bir aşamadır.

**Görünüm Durumu'nda hiçbir önemli bilgi depolamıyor olsam da bu güvenlik açığı konusunda endişe duymalı mıyım?**  
Evet. Web uygulamasında herhangi bir dosyada önemli bilgiler depoluyorsanız, bu güvenlik açığı, saldırganın ilgili dosyanın içeriğini açığa çıkarmak için kullanılabilir.

**web.config dosyasındaki verilerimin güvenliğini sağlamak için en iyi uygulamalar nelerdir?**  
En iyi uygulama, web.config dosyalarındaki önemli yapılandırma verilerini şifrelemektir. Yapılandırma bölümlerini şifreleme konusunda ayrıntılı bilgi için, [Yapılandırma Bölümlerini Şifreleme ve Şifresini Çözme](http://msdn.microsoft.com/en-us/library/zhhddkxy(vs.80).aspx) adlı MSDN makalesine bakın.

**Bu sorun hem ASP.NET Web Formu'nu hem de ASP.NET MVC Web uygulamalarını etkiliyor mu?**  
Evet. Genel olarak duyurulan yararlanma, Web Formları ve MVC de dahil tüm ASP.NET uygulamalarına yönelik olarak gerçekleştirilebilir. ASP.NET MVC hakkında ek bilgi için, [ASP.NET MVC'ye Genel Bakış](http://msdn.microsoft.com/en-us/library/dd381412.aspx) adlı MSDN makalesine bakın.

**ASP.NET platformu üzerine kurulan SharePoint ve Exchange gibi diğer uygulamalar bu sorundan etkileniyor mu?**  
ASP.NET platformunu kullanan tüm uygulamalar bu sorundan etkilenir. Bu uygulamaların yöneticileri, bu danışma belgesinde özetlenen önerileri uygulamalıdır.

**AES şifrelemesi yerine Üçlü DES şifrelemesine geçmek ASP.NET uygulamamın korunmasına yardımcı olur mu?**  
Hayır. Şifreleme ile ilgili olarak açıklanan güvenlik açığı, PKCS \#7 doldurma kullanan algoritmalar için bir istemciye şifreleme doldurma hatalarının açıklanmasıyla ilgilidir. Üçlü DES bu doldurma modunu AES ile paylaştığı için, Üçlü DES şifrelemesine geçmek bu güvenlik açığına karşı koruma sağlamaz.

**Üçlü DES (3DES) nedir?**  
Üçlü DES (3DES), her bir veri kesiminde şifreleme işlemlerinin üç yinelemesini kullanan bir Veri Şifreleme Standardı (DES) şifrelemesidir. Her yineleme, şifreleme için 56 bitlik bir anahtar kullanır ve böylece veriler için 168 bitlik şifreleme sağlanır. 3DES ek şifreleme hesaplamaları nedeniyle DES'den daha yavaştır, ancak DES'den çok daha güçlü koruma sağlar.

**Gelişmiş Şifreleme Standardı (AES) nedir?**  
Gelişmiş Şifreleme Standardı (AES), 2001 yılında ABD hükümeti tarafından uyarlanmış bir şifreleme biçimidir. AES, öncülü olan Veri Şifreleme Standardı'na (DES) göre daha güvenli şifreleme sağlar.

**PKCS \#7 doldurma nedir?**  
Ortak Anahtar Şifreleme Standardı (PKCS) \#7, dijital imzalar gibi şifreleme uygulanabilecek veriler için genel bir sözdizimidir. Bunun yanı sıra iletiye sertifikalar veya sertifika iptal listeleri ve zaman damgaları gibi diğer ileti özniteliklerini dağıtmak için kullanılan bir sözdizimi sağlar. PKCS \#7 doldurma dizesi, her biri eklenen toplam doldurma bayt sayısına eşit olan bir dizi bayttan oluşur.

**Bu sorun Oracle veritabanı ürünlerini etkiliyor mu?**  
Hayır. Bu sorun hiçbir veritabanı ürünüyle ilişkili değildir. "Doldurma bilgisi" saldırısının tanımında geçen "oracle" sözcüğü bir şifreleme terimidir.

**Güncelleştirme ne yapıyor?**  
Güncelleştirme, ASP.NET tarafından şifrelenen tüm verileri imzalayarak bu güvenlik açığını ortadan kaldırır.

**Bu güvenlik bülteni yayımlandığında bu güvenlik açığı genel olarak duyurulmuş muydu?**  
Evet. Bu güvenlik açığı genel olarak duyurulmuştur. Ortak Güvenlik Açığı ve Etkilenme (Common Vulnerability and Exposure) numarası olarak [CVE-2010-3332](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-3332) atanmıştır.

**Bu güvenlik bülteni yayımlandığında, Microsoft bu güvenlik açığından yararlanıldığına dair herhangi bir rapor almış mıydı?**  
Evet. Microsoft bu güvenlik açığından yararlanmaya çalışan sınırlı sayıda hedefe yönelik saldırı olduğunu bilmektedir.

[Uygulama Uyumluluğu Araç Takımı](http://www.microsoft.com/downloads/details.aspx?familyid=24da89e9-b581-47b0-b45e-492dd6da2971&displaylang=tr) ile birlikte gelen [Güncelleştirme Uyumluluğu Değerlendiricisi](http://technet2.microsoft.com/windowsvista/en/library/4279e239-37a4-44aa-aec5-4e70fe39f9de1033.mspx?mfr=true) bileşenlerini kullanarak, Windows güncelleştirmelerinin yüklü uygulamalara göre sınanması ve doğrulanması sürecini hızlandırabilirsiniz.

Uygulama Uyumluluğu Araç Takımı (ACT), çalışma ortamınıza Microsoft Windows Vista'yı, bir Microsoft Güvenlik Güncelleştirmesi'ni ya da Windows Internet Explorer'ın yeni bir sürümünü dağıtmadan önce uygulama uyumluluğu sorunlarını değerlendirmek ve etkilerini azaltmak için kullanılabilecek gerekli araçları ve belgeleri içerir.

-   **Microsoft Baseline Security Analyzer**

    Etkilenen bir sisteme güvenlik güncelleştirmesinin uygulandığını doğrulamak için Microsoft Baseline Security Analyzer (MBSA) aracını kullanabilirsiniz. Daha fazla bilgi için, bu bültenin önceki bölümünde yer alan **Algılama ve Dağıtım Araçları ve Kılavuzu** alt bölümüne bakın.

-   **Dosya Sürümü Doğrulaması**

    Microsoft Windows'un birçok sürümü olduğundan, aşağıdaki adımlar sizin sisteminizde farklı olabilir. Böyle bir durumda, bu adımları tamamlamak için ürün belgelerinize bakın.

    1.  **Başlat**'ı ve sonra **Ara**'yı tıklatın.
    2.  **Arama Sonuçları** bölmesinde, **Arama Yardımcısı**'nın altındaki **Tüm dosya ve klasörler**'i tıklatın.
    3.  **Dosya adının tümü veya bir parçası** kutusuna, uygun dosya bilgileri tablosundan bir dosya adı yazın ve **Ara**'yı tıklatın.
    4.  Dosya listesinde, uygun dosya bilgileri tablosundan bir dosya adını sağ tıklatın ve sonra **Özellikler**'i tıklatın.
        **Not** Sisteminizde yüklü işletim sisteminin veya programların sürümüne bağlı olarak, dosya bilgileri tablosunda listelenen dosyalardan bazıları yüklenmeyebilir.
    5.  **Sürüm** sekmesinde, sisteminize yüklenmiş olan dosyanın sürümünü uygun dosya bilgileri tablosunda belgelenen sürümle karşılaştırarak belirleyin.
        **Not** Dosya sürümü dışındaki öznitelikler yükleme sırasında değişebilir. Diğer dosya özniteliklerini dosya bilgileri tablosundaki bilgilerle karşılaştırmak, güncelleştirmenin uygulandığını doğrulama için desteklenen bir yöntem değildir. Ayrıca, belirli durumlarda, dosyalar kurulum sırasında yeniden adlandırılabilir. Dosya veya sürüm bilgileri yoksa, güncelleştirme kurulumunu kullanılabilen diğer yöntemlerden biriyle doğrulayın.

-   **Kayıt Defteri Anahtarı Doğrulaması**

    Bu güvenlik güncelleştirmesinde açıklanan dosyaların yüklenmiş olduğunu, bu bölümdeki **Başvuru Tablosu**'nda listelenen kayıt defteri anahtarlarını gözden geçirerek de doğrulayabilirsiniz.

**Güncelleştirmenin Uygulandığını Doğrulama**

-   **Microsoft Baseline Security Analyzer**

    Etkilenen bir sisteme güvenlik güncelleştirmesinin uygulandığını doğrulamak için Microsoft Baseline Security Analyzer (MBSA) aracını kullanabilirsiniz. Daha fazla bilgi için, bu bültenin önceki bölümünde yer alan Algılama ve Dağıtım Araçları ve Kılavuzu alt bölümüne bakın.

-   **Dosya Sürümü Doğrulaması**

    Microsoft Windows'un birçok sürümü olduğundan, aşağıdaki adımlar sizin sisteminizde farklı olabilir. Böyle bir durumda, bu adımları tamamlamak için ürün belgelerinize bakın.

    1.  **Başlat**'ı ve sonra **Ara**'yı tıklatın.
    2.  **Arama Sonuçları** bölmesinde, **Arama Yardımcısı**'nın altındaki **Tüm dosya ve klasörler**'i tıklatın.
    3.  **Dosya adının tümü veya bir parçası** kutusuna, uygun dosya bilgileri tablosundan bir dosya adı yazın ve **Ara**'yı tıklatın.
    4.  Dosya listesinde, uygun dosya bilgileri tablosundan bir dosya adını sağ tıklatın ve sonra **Özellikler**'i tıklatın.
        **Not** Sisteminizde yüklü işletim sisteminin veya programların sürümüne bağlı olarak, dosya bilgileri tablosunda listelenen dosyalardan bazıları yüklenmeyebilir.
    5.  **Sürüm** sekmesinde, sisteminize yüklenmiş olan dosyanın sürümünü uygun dosya bilgileri tablosunda belgelenen sürümle karşılaştırarak belirleyin.
        **Not** Dosya sürümü dışındaki öznitelikler yükleme sırasında değişebilir. Diğer dosya özniteliklerini dosya bilgileri tablosundaki bilgilerle karşılaştırmak, güncelleştirmenin uygulandığını doğrulama için desteklenen bir yöntem değildir. Ayrıca, belirli durumlarda, dosyalar kurulum sırasında yeniden adlandırılabilir. Dosya veya sürüm bilgileri yoksa, güncelleştirme kurulumunu kullanılabilen diğer yöntemlerden biriyle doğrulayın.

-   **Kayıt Defteri Anahtarı Doğrulaması**

    Bu güvenlik güncelleştirmesinde açıklanan dosyaların yüklenmiş olduğunu, bu bölümdeki **Başvuru Tablosu**'nda listelenen kayıt defteri anahtarlarını gözden geçirerek de doğrulayabilirsiniz.

    Bu kayıt defteri anahtarları yüklü dosyaların tam bir listesini içermeyebilir. Ayrıca bir yönetici veya OEM, bu güvenlik güncelleştirmesini Windows kurulum kaynak dosyalarıyla tümleştirdiğinde veya eklediğinde, bu kayıt defteri anahtarları doğru oluşturulmayabilir.

    5.  Son olarak, **Önceki Sürümler** sekmesini tıklatarak dosyanın önceki sürümünün dosya bilgilerini, yeni veya güncelleştirilmiş sürümünün dosya bilgileriyle de karşılaştırabilirsiniz.

    5.  Son olarak, **Önceki Sürümler** sekmesini tıklatarak dosyanın önceki sürümünün dosya bilgilerini, yeni veya güncelleştirilmiş sürümünün dosya bilgileriyle de karşılaştırabilirsiniz.
