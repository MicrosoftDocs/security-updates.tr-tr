Security Bulletin

Microsoft Güvenlik Bülteni MS10-041 - Önemli
============================================

Microsoft .NET Framework'teki Güvenlik Açığı Verilerde Değişiklik Yapılmasına Olanak Verebilir (981343)
-------------------------------------------------------------------------------------------------------

Yayım Tarihi: 8 Haziran 2010 Salı | Güncelleştirme Tarihi: 30 Haziran 2010 Çarşamba

**Sürüm:** 1.2

### Genel Bilgiler

#### Yürütmeyle İlgili Özet

Bu güvenlik güncelleştirmesi Microsoft .NET Framework'teki genel olarak duyurulan bir güvenlik açığını giderir. Güvenlik açığı, imzalanmış XML içeriğindeki verilerin fark edilmeden değişiklik yapılmasına olanak verebilir. Özel uygulamalardaki güvenlik etkisi, imzalanmış içeriğin ilgili uygulamada kullanılma şekline bağlıdır. İmzalanmış XML iletilerinin güvenli bir kanal (SSL gibi) üzerinden iletildiği senaryolar bu güvenlik açığından etkilenmez.

Güvenlik güncelleştirmesi Windows 2000, Windows XP, Windows Server 2003, Windows Vista, Windows Server 2008, Windows 7 ve Windows Server 2008 R2'nin tüm desteklenen sürümleri üzerinde Microsoft .NET Framework'ün tüm etkilenen sürümleri için Önemli olarak derecelendirilmiştir. Daha fazla bilgi için, bu bölümdeki **Etkilenen ve Etkilenmeyen Yazılımlar** alt bölümüne bakın.

Güvenlik güncelleştirmesi, Microsoft .NET Framework'teki XMLDsig önerisinin uygulanma biçimini değiştirerek bu güvenlik açığını giderir. Bu güvenlik açığı hakkında daha fazla bilgi için**,** bir sonraki bölüm olan **Güvenlik Açığı Bilgileri** altında, ilgili güvenlik açığı girdisinin Sık Sorulan Sorular (SSS) alt bölümüne bakın.

**Öneri.**  Müşterilerin büyük bir bölümü otomatik güncelleştirmeleri etkinleştirmiş durumdadır ve bu güvenlik güncelleştirmesi otomatik olarak karşıdan yüklenip kurulacağı için herhangi bir işlem yapmaları gerekmemektedir. Otomatik güncelleştirmeleri etkinleştirmemiş olan müşterilerin güncelleştirmeleri el ile denetlemeleri ve bu güncelleştirmeyi yüklemeleri gerekir. Otomatik güncelleştirmelerin özel yapılandırma seçenekleri hakkında bilgi için, bkz. [Microsoft Bilgi Bankası makalesi 294871](http://support.microsoft.com/kb/294871).

Yöneticiler ve kurumsal yüklemeler ya da bu güvenlik güncelleştirmesini el ile yüklemek isteyen son kullanıcılar için, Microsoft, müşterilerinin güncelleştirme yönetim yazılımlarını kullanarak veya [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) hizmeti aracılığıyla güncelleştirmeleri denetleyerek güncelleştirmeyi ilk fırsatta uygulamalarını önerir.

Bu bültenin sonrasında yer alan **Algılama ve Dağıtım Araçları ve Kılavuzu** bölümüne de bakın.

**Bilinen Sorunlar.** [Microsoft Bilgi Bankası makalesi 981343](http://support.microsoft.com/kb/981343), bu güvenlik güncelleştirmesini yükleyen müşterilerin karşılaşabileceği bilinen sorunları belgelemektedir. Bu makale ayrıca, bu sorunlar için önerilen çözümleri de içermektedir.

#### Etkilenen ve Etkilenmeyen Yazılımlar

Aşağıdaki yazılımlar, hangi sürümlerinin etkilendiğini belirlemek amacıyla sınanmıştır. Diğer sürümler destek ömrünü tamamlamıştır veya bu açıktan etkilenmemektedir. Yazılımınızın sürümünün destek ömrünü belirlemek için [Microsoft Destek Ömrü](http://go.microsoft.com/fwlink/?linkid=21742) Web sitesini ziyaret edin.

Bu Güvenlik Güncelleştirmesi İle İlgili Sık Sorulan Sorular (SSS)
-----------------------------------------------------------------

<span></span>
**Bu güvenlik bülteni 22 Haziran 2010'da neden yeniden düzenlendi? **
Microsoft bu bülteni, Microsoft .NET Framework 3.5 Service Pack 1'in yüklü olduğu sistemlerde Microsoft .NET Framework 3.5 için 8 Haziran 2010 tarihli güvenlik güncelleştirmesinin (KB982865) Microsoft Baseline Security Analyzer (MBSA) tarafından hatalı şekilde "yüklü" olarak listelenmesine neden olan bir sorunu düzelten algılama mantığı değişikliğini duyurmak üzere yeniden düzenledi. Bununla birlikte, Microsoft .NET Framework 2.0 Service Pack 2 ve Microsoft .NET Framework 3.5 Service Pack 1 güvenlik güncelleştirmesinin (KB979909) uygulanabilirliği MBSA tarafından doğru şekilde algılanmaktadır. Bu, ilk güncelleştirmenin içerdiği dosyaları etkilemeyen bir algılama değişikliğidir. Sistemlerini başarıyla güncelleştirmiş olan müşterilerin bu güncelleştirmeyi yeniden yüklemeleri gerekmez.

Bu algılama değişikliği, Windows Server Update Services (WSUS) kullanan müşterilerin KB982865 numaralı güncelleştirmeyi yeni bir güncelleştirme olarak görmelerine neden olabilir. Otomatik onaylama kuralları kullanmıyorsanız, güncelleştirmeyi el ile yeniden onaylamanız gerekebilir. Güncelleştirme yeniden onaylanmazsa, sisteme uygulanmıyor olsa da sunulmaya devam edebilir. Bu algılama değişikliği hakkında daha fazla bilgi için, bkz: [Microsoft Bilgi Bankası makalesi 894199](http://support.microsoft.com/kb/894199).

**Veri değiştirme güvenlik açığı nedir? **
Bilgi güvenliği açısından, veri değiştirme güvenlik açığı, verilerin kötü amaçlı olarak değiştirilmesine olanak verebilir. Örnek olarak, veritabanı gibi bir kaynakta depolanan kalıcı verilerin yetkisiz olarak değiştirilmesi ve Internet gibi açık bir ağ üzerinden iki bilgisayar arasında iletilen verilerin değiştirilmesi gösterilebilir. Bu güvenlik açığına özgü olarak, verileri dijital olarak imzalamak üzere etkin bir şifreleme imzası kullanılırsa, bu veriler imzalandıktan sonra değiştirilirse ve dijital imza doğrulanamazsa, bu durum verilerde değişiklik yapıldığı anlamına gelir. Veriler değiştirilmiş olmasına karşın imza yine de doğrulanabiliyorsa da verilerde değişiklik yapılmış demektir.

**Microsoft .NET Framework'ün hangi sürümünün yüklü olduğunu nasıl belirlerim?**  
Bir sistemde .NET Framework'ün birden çok sürümünü yükleyip çalıştırabilir ve bu sürümleri dilediğiniz sırayla yükleyebilirsiniz. .NET Framework'ün hangi sürümlerinin yüklü olduğunu belirlemenin çeşitli yolları vardır. Daha fazla bilgi için, [Microsoft Bilgi Bankası makalesi 318785](http://support.microsoft.com/kb/318785)'e veya [.NET Framework'ün Hangi Sürümünün Yüklü Olduğunu Belirleme](http://msdn.microsoft.com/en-us/library/y549e41e.aspx) adlı MSDN makalesine bakın.

**Desteklenen bazı işletim sistemlerinde Microsoft .NET Framework 3.5 ve Microsoft .NET Framework 3.5 Service Pack 1 neden etkileniyor?**  
Microsoft .NET Framework 3.5, Microsoft .NET Framework 2.0 Service Pack 1 ve Microsoft .NET Framework 3.0 Service Pack 1 alt bileşenlerini içermektedir. Microsoft .NET Framework 3.5 Service Pack 1, Microsoft .NET Framework 2.0 Service Pack 2 ve Microsoft .NET Framework 3.0 Service Pack 2 alt bileşenlerini içermektedir. Bazı işletim sistemlerinde, Microsoft .NET Framework 3.5 ve Microsoft .NET Framework 3.5 Service Pack 1, güvenlik açığı bulunan bu bileşenleri yükleyebilir.

**Sistemimde yüklü olan Microsoft .NET Framework sürümü bu bültende listelenmiyor. Yapılandırmam bu güvenlik açığından etkilenir mi?**  
Bu bültende listelenen etkilenen yazılımlar, hangi sürümlerinin etkilendiğini belirlemek amacıyla sınanmıştır. Diğer sürümler destek ömrünü tamamlamıştır veya desteklenmemektedir. Sistemlerinde Microsoft .NET Framework'ün desteklenmeyen bir sürümünün yüklü olduğu müşterilerin bu Microsoft .NET Framework sürümünü kaldırmaları ve daha yeni bir Microsoft .NET Framework sürümüne yükseltmeleri önerilir.

Microsoft .NET Framework, Denetim Masası'ndaki **Program Ekle veya Kaldır** aracı aracılığıyla kaldırılabilir. Microsoft .NET Framework'ün belirli sürümlerini kaldırma hakkında daha fazla bilgi için, bkz: [Microsoft Bilgi Bankası makalesi 320122](http://support.microsoft.com/kb/320112), [Microsoft Bilgi Bankası makalesi 824643](http://support.microsoft.com/kb/824643) ve [Microsoft Bilgi Bankası makalesi 908077](http://support.microsoft.com/kb/908077).

**Bu güncelleştirme, işlevlerde güvenlikle ilgili değişiklikler içeriyor mu?**  
Evet. Bu bültenin **Güvenlik Açığı Bilgileri** bölümünde listelenen değişikliklerin yanı sıra, bu güncelleştirme, bir saldırganın ASP.NET etkin Web sitelerinde varsayılan olarak etkinleştirilmiş durumdaki kapsamlı savunma önlemini atlamasına olanak verebilecek bir sorunu gidermek üzere ASP.NET istek doğrulama özelliğine yönelik kapsamlı bir savunma içerir.

Bu sorun, ASP.NET'teki istek doğrulama özelliğinin belirli bir karakter sırasını doğru denetlememesi nedeniyle ortaya çıkar. ASP.NET istek doğrulama özelliği, güvenilir olmayan giriş değişkenlerini kısıtlayan etkin bir doğrulama katmanının yerini alamaz. ASP.NET'in Web uygulamaları için sağladığı güvenlik özellikleri hakkında daha fazla bilgi edinmek isteyen geliştiriciler [ASP.NET'teki Yerleşik Özellikleri Kullanarak Web Saldırılarını Önleme](http://msdn2.microsoft.com/en-us/library/ms972969.aspx) adlı MSDN makalesine başvurabilirler.

Bir Web sitesinde güvenilir olmayan kullanıcı girişlerini kısıtlamaya yönelik etkin bir doğrulama katmanı yoksa, bu güvenlik açığından başarıyla yararlanan bir saldırgan, etkilenen Web sitesine ASP.NET içeriği de dahil rasgele içerik ekleyebilir. Bu kapsamlı savunma değişikliği, özel olarak bildirilen bu sorunun etkilerini azaltır.

**ASP.NET İstek Doğrulama nedir?**  
ASP.NET istek doğrulama özelliğindeki bu kapsamlı savunma değişikliği, ASP.NET çalışan Web sitelerinde temel giriş doğrulaması gerçekleştirir. Ancak ASP.NET istek doğrulama özelliği, güvenilir olmayan giriş değişkenlerini kısıtlayan etkin bir doğrulama katmanının yerini alamaz. ASP.NET'in Web uygulamaları için sağladığı güvenlik özellikleri hakkında daha fazla bilgi edinmek isteyen geliştiriciler [ASP.NET'teki Yerleşik Özellikleri Kullanarak Web Saldırılarını Önleme](http://msdn2.microsoft.com/en-us/library/ms972969.aspx) ve [ASP.NET İstek Doğrulama Kullanmayın](http://msdn.microsoft.com/en-us/library/ms998264.aspx) adlı MSDN makalesine başvurabilirler.

**ASP.NET Web uygulamamın bu kapsamlı savunma değişikliğinden etkilendiğini nasıl anlarım?**  
ASP.NET kullanılarak geliştirilen ve güvenilir olmayan tüm girdi değişkenlerini beklenen değer veya karakter aralığında kısıtlayan Web uygulamaları bu açıktan etkilenmez. ASP.NET Web uygulamalarının güvenliğini artırma konusunda daha fazla bilgi için, bkz. [Microsoft Bilgi Bankası makalesi 815155](http://support.microsoft.com/kb/815155).

**ASP.NET nedir?**  
ASP.NET, Microsoft .NET Framework'te bulunan ve geliştiricilerin Web uygulamaları ve XML Web Hizmetleri oluşturmasına olanak sağlayan bir grup teknolojidir.

Statik HTML ve komut dosyalarını bir arada kullanan geleneksel Web sayfalarından farklı olarak, ASP.NET derlenmiş ve olay kaynaklı sayfalar kullanır. ASP.NET Web tabanlı bir uygulama ortamı olduğu için, temel HTTP işlevselliğini sağlamak üzere arka planda bir Web sunucusunun bulunması gerektiğinden Internet Information Services'ın (IIS) üzerinde çalışır. Daha fazla bilgi için, bkz: [Resmi Microsoft ASP.NET Sitesi](http://www.asp.net/).

**Bu sorun neden kapsamlı bir savunma önlemi olarak gideriliyor?**  
ASP.NET istek doğrulama özelliği, geliştiricinin kendi giriş doğrulamasına ek bir önlem olarak kapsamlı bir savunma değişikliği niteliğindedir. Belirli bir uygulama için uygun içeriğin ne olduğunu yalnızca geliştirici belirleyebilir. Kapsamlı savunma özellikleri etkin olarak kullanılacak biçimde tasarlanmazlar; ancak bu tür birden çok katman olması, söz konusu sistemin güvenliğinin saldırganlar tarafından aşılmasının engellenmesine yardımcı olacaktır. Bu nedenle, ASP.NET istek doğrulama özelliğinde belirli bir karakter sırası eksikse, bu sorunun en yüksek güvenlik etkisi, bu özellik devre dışı bırakıldığında karşılaşılacak etkiden daha yüksek olamaz.

**Kapsamlı savunma nedir?**  
Bilgi güvenliğinde kapsamlı savunma, bir ağın veya sistemin güvenliğinin saldırganlar tarafından aşılmasını önlemeye yardımcı olmak üzere birden çok savunma katmanı kullanılan bir yaklaşım anlamına gelir.

XML İmzasında HMAC Kesilmesi Nedeniyle Kimlik Doğrulamasını Atlama Güvenlik Açığı - CVE-2009-0217
-------------------------------------------------------------------------------------------------

<span></span>
Microsoft .NET Framework'te, bir saldırganın imzalanmış XML içeriğindeki verilerde fark edilmeden değişiklik yapmasına olanak verebilecek bir veri değiştirme güvenlik açığı bulunmaktadır. Özel uygulamalardaki güvenlik etkisi, ilgili kullanım senaryosuna bağlıdır. İmzalanmış XML iletilerinin güvenli bir kanal (SSL gibi) üzerinden iletildiği senaryolar bu güvenlik açığından etkilenmez.

Bu güvenlik açığını Ortak Güvenlik Açıkları ve Etkilenmeler listesinde standart girdi olarak görüntülemek için, bkz. [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217).

#### XML İmzasında HMAC Kesilmesi Nedeniyle Kimlik Doğrulamasını Atlama Güvenlik Açığı - CVE-2009-0217 için Azaltıcı Etkenler

Azaltıcı etken, varolan bir durumda bulunan ve güvenlik açığından yararlanılmasının önem derecesini azaltabilecek bir ayar, genel yapılandırma ya da en iyi uygulama anlamına gelir. Sizin durumunuzda aşağıdaki azaltıcı etkenler işinize yarayabilir:

-   Microsoft Office, bu veri değiştirme güvenlik açığından yararlanmaya çalışan bir saldırganın kullanamayacağı şekilde XML imzası denetimi gerçekleştirir.

#### XML İmzasında HMAC Kesilmesi Nedeniyle Kimlik Doğrulamasını Atlama Güvenlik Açığı - CVE-2009-0217 için Geçici Çözümler

Geçici çözüm, arka plandaki güvenlik açığını gidermeyen ama güvenlik açığı uygulanmadan önceki bilinen saldırı vektörlerinin engellenmesine yardımcı olan bir ayar veya yapılandırma değişikliği anlamına gelir. Microsoft aşağıdaki geçici çözümleri sınamıştır ve bir geçici çözümün işlevselliği azaltıp azaltmadığı ilgili açıklamada belirtilmektedir:

-   **Önemli bilgileri iletmek için güvenli bir kanal kullanın**

    Uygulamanız önemli XML verileri veya uygulamanın güvenliği açısından önemli XML verileri iletiyorsa, bu bilgileri iletmek için güvenli bir kanal kullanın. Örneğin, Internet Information Server kullanıyorsanız, önemli veya güvenlik açısından kritik uygulamalar için SSL kullanılmasını gerektirin.

#### XML İmzasında HMAC Kesilmesi Nedeniyle Kimlik Doğrulamasını Atlama Güvenlik Açığı - CVE-2009-0217 Hakkında SSS

**Bu güvenlik açığının kapsamı nedir?**  
Bu güvenlik açığı, bir şifreleme imzasını atlamak için kullanılabilecek bir veri değiştirme güvenlik açığıdır.

**Güvenlik açığına ne neden olmaktadır?**  
Güvenlik açığı, W3C XML Signature Syntax and Processing (XMLDsig) önerisinin Microsoft .NET Framework'te uygulanma biçimi nedeniyle ortaya çıkar. XMLDsig hakkında daha fazla bilgi için, [XML Signature Syntax and Processing (İkinci Sürüm)](http://www.w3.org/tr/xmldsig-core/) adlı W3C Önerisi'ne veya [XML İmzası İkinci Sürüm Hatası](http://www.w3.org/2008/06/xmldsigcore-errata.html) içindeki "E03 HMAC kesilmesi (CVE-2009-0217)" bölümüne bakın.

**Microsoft .NET Framework nedir?**  
Microsoft .NET Framework, yazılım uygulamaları ve Web hizmetleri oluşturulmasına ve çalıştırılmasına olanak veren bir Microsoft Windows işletim sistemi bileşenidir. Web hizmetleri ve Web uygulamaları (ASP.NET), veri erişimi (ADO.NET), akıllı istemci uygulamaları (Windows Formları) ve daha birçok teknoloji içerir.

**HMAC nedir?**  
Gönderenin ve alıcının bir gizli anahtarı paylaşması koşuluyla, güvenli olmayan kanal üzerinden gönderilen bir iletinin bütünlüğü ve orijinalliği doğrulanarak iletide değişiklik yapılmış olup olmadığını belirlemek üzere karma tabanlı bir İleti Kimlik Doğrulama Kodu (HMAC) kullanılabilir.

**XML nedir?**  
Genişletilebilir Biçimlendirme Dili veya kısaca XML, yapılandırılmış verileri açıklamaya yönelik bir biçim sağlayan bir biçimlendirme dilidir. Genişletilebilir Biçimlendirme Dili (XML) bir World Wide Web Consortium (W3C) belirtimi ve bir Standard Generalized Markup Language (SGML) alt kümesidir. XML hakkında daha fazla bilgi için, [XML giriş sayfası](http://msdn.microsoft.com/xml/) adlı MSDN Veri Geliştirici Merkezi sayfasına veya [Genişletilebilir Biçimlendirme Dili (XML) 1.0 (Beşinci Sürüm)](http://www.w3.org/tr/rec-xml/) adlı W3C Önerisi'ne bakın.

**Bir saldırgan bu güvenlik açığından yararlanarak neler yapabilir?**  
Bu güvenlik açığından başarıyla yararlanan bir saldırgan belirli şifreleme imzalarını atlayarak, imzalanmış XML içeriğinde alıcının fark edemeyeceği şekilde değişiklik yapabilir. İletinin özgün içeriği tümüyle değiştirilirse, bu durum kimlik sahtekarlığı olarak da nitelendirilebilir. Bir geliştiricinin HMAC imzalı XML içeriğini kullanan bir uygulama yazdığı durumlarda, güvenlik etkisi, bu uygulamaya özgü kullanım senaryosuna bağlıdır.

**Bir saldırgan güvenlik açığından nasıl yararlanabilir?**  
Saldırganın bu güvenlik açığından etkilenen bir sisteme özel hazırlanmış XML içeriği göndermesi gerekir. Microsoft Office, bir saldırganın yararlanamayacağı şekilde XML imzası denetimi gerçekleştirir.

**Bu güvenlik açığından birincil derecede hangi sistemler etkileniyor?**  
Şifreleme işlemleri için karma tabanlı İleti Kimlik Doğrulama Kodu (HMAC) kullanılan sistemler veya uygulamalar birincil derecede risk altındadır. Microsoft Office, bu güvenlik açığından yararlanmaya çalışan bir saldırganın kullanamayacağı şekilde XML imzası denetimi gerçekleştirir.

**Güncelleştirme ne yapıyor?**  
Güncelleştirme, Microsoft .NET Framework'te XMLDsig önerisinin uygulanma biçimini değiştirerek bu güvenlik açığını giderir.

**Bu güvenlik bülteni yayımlandığında bu güvenlik açığı genel olarak duyurulmuş muydu?**  
Evet. Bu güvenlik açığı genel olarak duyurulmuştur. Ortak Güvenlik Açığı ve Etkilenme (Common Vulnerability and Exposure) numarası olarak [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217) atanmıştır.

**Bu güvenlik bülteni yayımlandığında, Microsoft bu güvenlik açığından yararlanıldığına dair herhangi bir rapor almış mıydı?**  
Hayır. Bu güvenlik bülteni ilk yayımlandığında, Microsoft bu güvenlik açığının müşterilere saldırmak üzere genel olarak kullanıldığına dair herhangi bir bilgi almamıştı.

    2.  **Arama Sonuçları** bölmesinde, **Arama Yardımcısı**'nın altındaki **Tüm dosya ve klasörler**'i tıklatın.

    Microsoft Windows'un birçok sürümü olduğundan, aşağıdaki adımlar sizin sisteminizde farklı olabilir. Böyle bir durumda, bu adımları tamamlamak için ürün belgelerinize bakın.

    2.  **Arama Sonuçları** bölmesinde, **Arama Yardımcısı**'nın altındaki **Tüm dosya ve klasörler**'i tıklatın.
    3.  **Dosya adının tümü veya bir parçası** kutusuna, uygun dosya bilgileri tablosundan bir dosya adı yazın ve **Ara**'yı tıklatın.
    4.  Dosya listesinde, uygun dosya bilgileri tablosundan bir dosya adını sağ tıklatın ve sonra **Özellikler**'i tıklatın.

    5.  **Sürüm** sekmesinde, sisteminize yüklenmiş olan dosyanın sürümünü uygun dosya bilgileri tablosunda belgelenen sürümle karşılaştırarak belirleyin.

Diğer dosya özniteliklerini dosya bilgileri tablosundaki bilgilerle karşılaştırmak, güncelleştirmenin uygulandığını doğrulama için desteklenen bir yöntem değildir.
